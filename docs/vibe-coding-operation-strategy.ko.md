# 바이브 코딩과 Ontology LLM Wiki 운영 전략

## 결론

가장 효과적인 전략은 LLM Wiki를 바이브 코딩의 **기억 장치이자 품질 검사 장치**로 쓰는 것이다.

바이브 코딩은 자연어로 빠르게 만들고 고치는 방식이다. 장점은 실행 속도이고, 약점은 맥락 휘발, 검증 부족, 과한 즉흥성이다.

Ontology LLM Wiki는 이 약점을 보완한다.

```text
바이브 코딩 = 빠르게 만든다
LLM Wiki = 만든 이유와 배운 점을 축적한다
Ontology = 축적된 지식의 관계와 오류를 검사한다
```

즉, 이 시스템은 "AI로 코딩하기"를 넘어서 **AI와 함께 일하는 방식 자체를 누적하는 운영체제**가 된다.

## 1. 핵심 운영 원칙

### 작업 전에는 Query

새 작업을 시작하기 전에 AI에게 바로 만들라고 하지 않는다. 먼저 위키를 읽게 한다.

```text
wiki/index.md와 관련 concept/workflow 페이지를 먼저 읽고,
이번 작업과 관련된 기존 결정, 주의사항, 반복 패턴을 요약해줘.
그 다음 오늘 할 작업의 성공 기준을 제안해줘.
```

목적:

- 예전 결정을 반복해서 설명하지 않기
- 과거 실수를 다시 하지 않기
- 오늘 작업의 성공 기준을 먼저 고정하기

### 작업 중에는 Vibe Coding

자연어로 원하는 결과를 설명하고, AI가 파일을 만들거나 수정하게 한다.

단, 명령은 "코드를 이렇게 짜"보다 "성공 기준"으로 준다.

나쁜 예:

```text
이 기능 만들어줘.
```

좋은 예:

```text
초보자가 ZIP을 받아 10분 안에 첫 raw 자료를 ingest할 수 있어야 해.
필요한 문서, 템플릿, 검사 명령을 추가하고,
마지막에 reasoner/lint가 통과하는지 확인해줘.
```

### 작업 후에는 Ingest

작업이 끝나면 결과를 그냥 두지 않는다. 이번 작업에서 배운 것을 위키에 저장한다.

```text
이번 작업에서 새로 생긴 결정, 반복 가능한 절차, 실패/주의사항을 wiki에 반영해줘.
필요하면 ontology/abox.json에 concept, workflow, claim을 추가하고 reasoner를 실행해줘.
```

목적:

- 다음 작업의 시작점이 좋아짐
- 좋은 프롬프트와 판단 기준이 재사용됨
- 프로젝트가 커질수록 AI가 더 똑똑하게 일함

### 정기적으로 Lint

계속 만들기만 하면 위키도 지저분해진다. 그래서 주기적으로 검사한다.

```text
wiki를 lint 해줘.
고립 문서, 출처 없는 주장, 모순 후보, 오래된 정보, ontology domain/range 오류를 찾아줘.
수정할 것과 보류할 것을 분리해서 제안해줘.
```

## 2. 바이브 코딩 루프

이 시스템의 기본 루프는 다음이다.

```text
1. Query: 기존 지식 불러오기
2. Plan: 오늘 성공 기준 정하기
3. Build: 바이브 코딩으로 만들기
4. Verify: 실행/테스트/reasoner/lint 확인
5. Ingest: 배운 점과 결정을 wiki에 저장
6. Evolve: 반복되는 패턴을 schema/ontology로 승격
```

중요한 점은 5번이다. 대부분의 바이브 코딩은 Build에서 끝난다. 이 프로젝트는 Build 이후에 Ingest를 강제해서 지식이 복리로 쌓이게 만든다.

## 3. 사람과 AI의 역할 분리

### 사람의 역할

- 어떤 자료를 넣을지 고른다.
- 오늘 만들 결과의 성공 기준을 정한다.
- AI가 만든 결과를 보고 승인/보류를 결정한다.
- 도메인에서 중요한 개념 이름을 확정한다.

### AI의 역할

- raw 자료를 읽고 wiki로 정리한다.
- 반복 개념을 찾아 연결한다.
- 파일을 만들고 수정한다.
- 오류 메시지를 보고 고친다.
- 작업 후 배운 점을 다시 wiki에 저장한다.

### Ontology의 역할

- 개념 종류를 정리한다.
- 관계가 말이 되는지 확인한다.
- 잘못된 연결을 잡는다.
- 자동으로 추론 가능한 관계를 만든다.

## 4. 초보자용 하루 운영법

### 시작 5분

```text
오늘 작업하기 전에 wiki/index.md를 읽고,
내가 지금 이어서 봐야 할 문서 3개와 오늘의 추천 작업 1개를 알려줘.
```

### 작업 30분

```text
추천 작업을 기준으로 필요한 파일을 수정해줘.
초보자가 이해할 수 있는 구조를 유지하고,
작업 후 검증 명령까지 실행해줘.
```

### 마무리 10분

```text
오늘 작업에서 생긴 결정, 배운 점, 다음에 반복할 절차를 wiki에 저장해줘.
ontology에 추가할 만한 concept/workflow/claim이 있으면 제안하고,
reasoner와 lint를 실행해줘.
```

이 세 문장만 지켜도 초보자는 "매번 새로 시작하는 AI 사용"에서 벗어날 수 있다.

## 5. 지속 디벨롭 전략

### 1단계: Markdown-first

처음에는 파일과 폴더만 쓴다.

- raw 자료 5개 이하
- wiki concept 10개 이하
- ontology class는 기본값 유지
- reasoner 오류 0개 유지

목표는 거대한 시스템이 아니라 "계속 쓰는 습관"이다.

### 2단계: Schema 강화

같은 수정 요청이 3번 반복되면 schema로 승격한다.

예:

- 매번 source page 형식을 고친다 -> `schema/page-template.md` 수정
- 매번 출처 표기를 빼먹는다 -> `schema/wiki-rules.md`에 필수 규칙 추가
- 매번 같은 검증을 한다 -> `CLAUDE.md` 체크리스트 강화

### 3단계: Ontology 강화

같은 관계가 2개 이상의 자료에서 반복되면 formal relation으로 승격한다.

예:

```text
Workflow uses_tool Tool
Source supports_claim Claim
WikiPage derived_from Source
```

반대로 한 번만 나온 관계는 성급하게 ontology에 넣지 않는다. Markdown 링크로 충분하다.

### 4단계: 자동화 추가

수작업이 반복될 때만 자동화를 붙인다.

추가 순서:

1. 검사 스크립트
2. 패키징 스크립트
3. ingest helper
4. Obsidian 플러그인/MCP
5. RAG/vector DB
6. graph DB

RAG나 graph DB는 처음부터 붙이지 않는다. 위키가 실제로 커져서 검색이 병목이 될 때 붙인다.

## 6. 기대효과

### 개인에게

- AI에게 매번 배경 설명을 반복하지 않아도 된다.
- 공부/업무/콘텐츠/프로젝트 노하우가 한 곳에 쌓인다.
- 내가 어떤 결정을 왜 했는지 추적할 수 있다.
- 바이브 코딩 결과물이 일회성 실험이 아니라 자산이 된다.

### 프로젝트에

- 개발 속도와 지식 품질이 함께 오른다.
- 새 기능을 만들 때 기존 결정과 충돌하는지 확인할 수 있다.
- 문서, 코드, 판단 기준이 따로 놀지 않는다.
- 초보자도 "무엇을 물어봐야 하는지"부터 안내받을 수 있다.

### 팀/배포 관점에서

- Git 또는 ZIP으로 같은 운영 구조를 배포할 수 있다.
- 각자 raw 자료는 달라도 schema와 ontology는 공유할 수 있다.
- 팀의 반복 판단이 위키와 온톨로지에 쌓인다.
- 나중에 RAG, MCP, graph DB로 확장할 때 기준 구조가 이미 있다.

## 7. 운영 지표

이 시스템이 잘 운영되는지는 코드 줄 수가 아니라 아래 지표로 본다.

- 새 작업 전 wiki query를 했는가?
- 작업 후 ingest를 했는가?
- reasoner 오류가 0개인가?
- wiki lint 경고가 줄고 있는가?
- 반복되는 프롬프트가 schema로 승격됐는가?
- 반복되는 관계가 ontology로 승격됐는가?
- 초보자가 다음 행동을 스스로 고를 수 있는가?

## 8. 한 문장 전략

```text
바이브 코딩으로 빠르게 실행하고,
LLM Wiki로 실행의 기억을 남기고,
Ontology로 그 기억의 관계와 품질을 검사한다.
```

이 세 가지가 함께 돌아갈 때, AI 사용은 단순 자동완성이 아니라 계속 성장하는 개인/팀 지식 시스템이 된다.
